# TLS/SSL

---

## 1. SSL (Secure Sockets Layer)

- 1995년 Netscape에서 개발한 암호화 기반 인터넷 보안 프로토콜.
- 웹에서의 데이터 전송을 암호화하여 개인정보 보호, 인증, 데이터 무결성을 보장.
- SSL 3.0 이후로 업데이트되지 않음.
- 현재는 TLS로 대체되며, SSL은 알려진 취약성이 있어 사용 중단을 권장.
- SSL/TLS 사용 웹사이트는 HTTPS URL을 사용.

---

## 2. TLS (Transport Layer Security)

- SSL의 후속 버전으로, SSL 3.0과 첫 TLS 버전 간 차이는 적음.
- SSL의 소유권이 Netscape에서 변경되어 TLS라는 명칭으로 발전.
- 사실상 SSL의 업데이트된 버전으로, SSL을 제공하는 업체도 실제로는 TLS 암호화를 사용.

---

## 3. SSL/TLS의 작동 방식

- SSL/TLS는 데이터 암호화를 통해 가로채기 어려운 형태로 전송.
- 클라이언트와 서버 간 핸드셰이크로 인증 진행.
- 데이터 무결성 보장을 위해 디지털 서명을 사용하여 데이터 변경 여부를 확인.

---

## 4. SSL/TLS 핸드셰이크 과정 (RSA 키 교환 알고리즘 예시)

1. 클라이언트 -> 서버: TLS 버전, 암호화 알고리즘, 무작위 바이트 문자열 전송.
2. 서버 -> 클라이언트: SSL 인증서, 암호화 알고리즘, 서버가 생성한 무작위 바이트 문자열 전송.
3. 인증: 클라이언트가 서버의 인증서를 발급 기관에 검증.
4. 예비 마스터 암호: 클라이언트가 서버로 전송한 premaster secret 키 암호화.
5. 개인 키 복호화: 서버가 개인 키를 사용해 premaster secret 키 복호화.
6. 세션 키 생성: 양측이 세션 키를 동일하게 생성.
7. 완료 메시지: 클라이언트와 서버가 세션 키로 암호화된 완료 메시지 전송.
8. 핸드셰이크 완료: 핸드셰이크가 완료되고, 세션 키로 실제 통신 진행.

---

## 5. SSL/TLS 인증서

- SSL 인증서(또는 TLS 인증서)는 암호화를 위해 필요.
- 인증서에는 공개 키가 포함되어 있으며, 이를 통해 암호화가 가능.
- 인증서 발급 기관(CA, Certificate Authority)이 공개 키를 인증.
- 클라이언트는 공개 키로 암호화하여 서버로 전송하고, 서버는 개인 키로 복호화.

---

## 6. 인증서 종류 및 유형

- 적용되는 도메인 개수에 따른 유형
    - 단일 도메인 (Single Domain): 하나의 도메인에만 적용.
    - 와일드카드 (Wildcard): 하위 도메인까지 포함되는 인증서.
    - 멀티 도메인 (Multi-Domain): 여러 관련 없는 도메인에 적용.
- 유효성 검사 수준에 따른 유형
    - 도메인 유효성 검사 (Domain Validation, DV): 가장 낮은 수준으로, 도메인 관리만 증명.
    - 조직 유효성 검사 (Organization Validation, OV): 기업 확인 및 증명.
    - 확장 유효성 검사 (Extended Validation, EV): 법적 사업자 등록 및 주소 확인을 포함, 가장 높은 신뢰도 제공.
