# GET과 POST

---

## 1. HTTP Method란?

- 정의
  - HTTP Method는 클라이언트-서버 구조에서 요청(request)과 응답(response)이 이루어지는 방식.
  - 다시 말해, 클라이언트가 서버에 어떤 동작을 수행할지 지시하는 방법이다.

- 왜 사용하는가?
  - URI는 리소스 식별에만 집중하고,
  - HTTP 메서드는 해당 리소스를 어떻게 다룰지 명시하여
  - 리소스와 동작을 분리하기 위함.

### 정리

| 항목            | 설명 |
|-----------------|------|
| 목적            | 서버에 특정 동작을 요청하기 위해 사용 |
| 역할            | 리소스를 다루는 방식을 명확히 구분 (조회, 생성, 수정, 삭제 등) |
| 장점            | RESTful 구조에서 명확한 설계 가능, URI 간결성 유지 |
| 주요 사용 메서드 | GET, POST, PUT, DELETE, PATCH |

---

## 2. GET 방식

- 사용 목적: 서버로부터 정보를 요청할 때 사용
- 데이터 전달 방식: 요청 데이터를 URL의 쿼리 스트링에 포함시켜 전송
  - 예: `www.example.com/resources?name1=value1&name2=value2`
- 데이터 위치: HTTP 패킷의 헤더
- 특징:
    - URL에 노출되므로 민감한 정보 전송에는 부적합
    - 브라우저 히스토리에 저장됨
    - 요청 길이에 제한 있음
    - 캐시 가능 → 정적 콘텐츠에 적합

---

## 3. POST 방식

- 사용 목적: 서버에 정보를 생성하거나 업데이트할 때 사용
- 데이터 전달 방식: 요청 데이터를 HTTP 메시지의 바디에 포함시켜 전송
- 데이터 위치: HTTP 패킷의 바디
- 특징:
    - URL에 노출되지 않아 보안성 높음
    - 브라우저 히스토리에 저장되지 않음
    - 데이터 길이 제한 없음
    - 캐시되지 않음

---

## 4.? GET과 POST의 주요 차이점

| 항목             | GET 방식                                       | POST 방식                                      |
|------------------|-----------------------------------------------|-----------------------------------------------|
| 사용 목적        | 데이터 조회 (SELECT)                          | 데이터 생성/업데이트 (CREATE/UPDATE)          |
| 데이터 위치      | URL의 쿼리 스트링                             | HTTP 바디                                     |
| 보안성           | 낮음 (데이터가 URL에 노출됨)                  | 높음 (데이터가 바디에 포함됨)                 |
| 히스토리 저장    | 저장됨                                        | 저장되지 않음                                 |
| 데이터 길이 제한 | 있음                                          | 없음                                          |
| 캐시 가능 여부   | 가능                                          | 불가능                                        |
| 멱등성           | 있음 (여러 번 요청해도 결과 동일)             | 없음 (여러 번 요청 시 결과 달라질 수 있음)    |
- 멱등성(idempotent): 동일한 요청을 여러 번 수행해도 결과가 변하지 않는 성질

---

## 5. 다른 HTTP 메소드

### PUT 방식

- 사용 목적: 리소스를 완전히 대체하거나 수정
- 데이터 전달 방식: 요청 데이터를 HTTP 메시지의 바디에 포함시켜 전송
- 데이터 위치: HTTP 패킷의 바디
- 특징:
    - 클라이언트가 리소스의 위치를 알고 있어야 함
    - 기존 리소스를 완전히 덮어씀
    - 멱등성을 가짐 (여러 번 요청해도 결과 동일)

### PATCH 방식

- 사용 목적: 리소스의 일부를 수정
- 데이터 전달 방식: 수정할 부분만을 HTTP 메시지의 바디에 포함시켜 전송
- 데이터 위치: HTTP 패킷의 바디
- 특징:
    - 리소스의 일부만 변경
    - 멱등성을 가지지 않음 (여러 번 요청 시 결과 달라질 수 있음)
    - 일부 서버에서는 지원하지 않을 수 있음

### DELETE 방식

- 사용 목적: 리소스를 삭제
- 데이터 전달 방식: 일반적으로 바디 없이 요청
- 데이터 위치: HTTP 패킷의 헤더
- 특징:
    - 지정된 리소스를 삭제
    - 멱등성을 가짐 (여러 번 요청해도 결과 동일)

### HEAD 방식

- 사용 목적: 리소스의 헤더 정보만 요청
- 데이터 전달 방식: 요청 데이터를 URL의 쿼리 스트링에 포함시켜 전송
- 데이터 위치: HTTP 패킷의 헤더
- 특징:
    - GET과 동일하지만 응답 본문은 없음
    - 리소스의 메타데이터 확인에 사용
    - 멱등성을 가짐 (여러 번 요청해도 결과 동일)

### OPTIONS 방식

- 사용 목적: 서버가 지원하는 메서드 및 옵션 확인
- 데이터 전달 방식: 일반적으로 바디 없이 요청
- 데이터 위치: HTTP 패킷의 헤더
- 특징:
    - 특정 리소스에 대해 지원되는 메서드 확인
    - 주로 CORS 설정 확인에 사용
    - 멱등성을 가짐 (여러 번 요청해도 결과 동일)

### CONNECT 방식

- 사용 목적: 터널을 설정하여 서버에 연결
- 데이터 전달 방식: 일반적으로 바디 없이 요청
- 데이터 위치: HTTP 패킷의 헤더
- 특징:
    - 주로 프록시 서버를 통해 SSL 터널링에 사용
    - 멱등성을 가지지 않음 (여러 번 요청 시 결과 달라질 수 있음)

### TRACE 방식

- 사용 목적: 요청 경로를 따라 메시지를 루프백 테스트
- 데이터 전달 방식: 요청 데이터를 그대로 전송
- 데이터 위치: HTTP 패킷의 헤더
- 특징:
    - 요청이 서버에 도달하는 경로를 확인
    - 디버깅 목적에 사용
    - 멱등성을 가짐 (여러 번 요청해도 결과 동일)
