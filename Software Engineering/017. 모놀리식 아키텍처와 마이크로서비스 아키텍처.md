# 모놀리식 아키텍처와 마이크로서비스 아키텍쳐

---

## 1. 모놀리식 아키텍처 (Monolithic Architecture)

### 정의

- 하나의 통합된 코드베이스로 모든 기능이 포함된 전통적인 아키텍처 스타일

### 장점

- 단일 코드베이스로 유지보수가 간편함
- 작은 팀이나 프로젝트에서 빠른 개발 가능
- 통합 및 테스트가 간단함
- 인프라 구조의 복잡성이 낮아 운영 비용 절감
- 기술 스택을 통일하기 쉬움
- 디버깅 및 최적화가 간단함

### 단점

- 프로젝트가 커질수록 확장과 배포가 어려움
- 기술 스택이 제한되어 새로운 기술 도입이 어려움
- 코드베이스가 커지며 복잡성 증가
- 대규모 팀 작업 시 충돌 가능성 높음

### 적합한 경우

- 소규모 프로젝트나 MVP 개발 시
- 복잡하지 않은 비즈니스 로직
- 변경이 적은 시스템

---

## 2. 마이크로서비스 아키텍처 (MSA, Microservices Architecture)

### 정의

- 애플리케이션을 여러 작고 독립적인 서비스로 나누는 아키텍처 스타일

### 장점

- 독립적인 개발, 배포, 확장 가능
- 각 서비스마다 최적의 기술 스택 선택 가능
- 특정 서비스 유지보수 용이
- 빠른 배포 주기 가능
- 특정 서비스만 확장 가능 → 자원 효율적 사용

### 단점

- 서비스 간 통신 및 관리 복잡
- 데이터 일관성 유지 어려움
- 통합 및 종단 간 테스트 어려움
- 전체 시스템 복잡도 증가
- 모놀리식에서의 전환 비용 및 시간 소요
- 분산 시스템 문제 처리 필요

### 적합한 경우

- 대규모 및 복잡한 시스템
- 독립적인 팀이 존재할 경우
- 유연한 확장이 요구될 경우

---

## 3. 모놀리식 아키텍처 vs 마이크로서비스 아키텍처

| 구분         | 모놀리식 아키텍처                     | 마이크로서비스 아키텍처                |
|--------------|--------------------------------------|----------------------------------------|
| 구조         | 단일 애플리케이션으로 기능 통합      | 독립적인 서비스로 구성                |
| 배포         | 전체 애플리케이션을 한 번에 배포     | 각 서비스를 독립적으로 배포           |
| 기술 다양성  | 동일한 기술 스택 사용                | 각 서비스마다 기술 스택 선택 가능     |
| 유연성       | 전체를 확장해야 함                   | 특정 서비스만 확장 가능               |
| 지연 시간    | 네트워크 대기시간 없음               | 서비스 간 통신으로 인한 지연 발생     |

- 아키텍처 선택은 프로젝트 특성, 규모, 팀의 역량 등을 고려해 결정하는 것이 중요합니다.
