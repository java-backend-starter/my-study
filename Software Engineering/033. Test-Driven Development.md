# Test Drive Development

---

## 1.TDD의 개요

- TDD(Test Driven Development)는 테스트 주도 개발으로, 테스트 코드를 먼저 작성하고 이를 통과하는 코드를 작성하는 개발 방법론이다. 
- 이 방법론은 애자일 개발의 일종으로, eXtreme Programming(XP)의 'Test-First' 개념에 기반하고 있다.

---

## 2. TDD의 핵심 단계

TDD는 3가지 핵심 단계로 이루어져 있다. 이 과정을 Red-Green-Blue라고 부른다:

1. Red 단계 (실패하는 테스트 작성)
    - 테스트 코드를 먼저 작성하는 단계이다. 
    - 이때 테스트는 실패한다. 왜냐하면, 아직 해당 기능을 구현하지 않았기 때문이다.
    - 예를 들어, "생년월일을 입력받아 나이를 출력하는 기능"을 테스트한다고 할 때, 이 단계에서는 "2015년과 2018년을 입력하면 3이 나온다"라는 테스트를 작성한다. 이 테스트는 기능이 없으므로 실패한다.
2. Green 단계 (테스트 통과하는 코드 작성)
    - 테스트를 통과하는 최소한의 코드를 작성하는 단계이다. 
    - Red 단계에서 작성한 실패하는 테스트를 통과시킬 수 있도록 최소한의 코드만을 구현한다.
    - 예를 들어, "2018 - 2015"의 뺄셈을 구현하여 3이라는 결과를 반환하는 코드를 작성한다.
3. Blue 단계 (리팩토링)
    - 테스트가 통과하면, 이제 코드를 개선하고 최적화하는 단계이다. 
    - 여기서는 중복된 코드 제거, 가독성 향상, 성능 개선 등을 수행한다.
    - 이 단계에서 중요한 점은, 테스트를 통과한 상태에서만 리팩토링을 수행한다는 것이다.

---

## 3. TDD의 주요 이점

1. 디버깅 시간 단축
    - TDD에서는 작은 단위로 테스트를 진행하기 때문에, 코드 작성 후 바로 버그를 찾고 수정할 수 있다. 예를 들어, 특정 입력 값에서 문제가 발생했다면, 문제를 쉽게 추적하여 특정 레이어(UI, 비즈니스 로직 등)에서 발생한 오류를 정확히 찾을 수 있다.
2. 빠른 피드백
    - TDD는 코드가 완성되기 전에 테스트를 먼저 작성하기 때문에, 기능 단위로 진행하며 빠른 피드백을 받을 수 있다. 이는 개발자가 코드의 품질을 지속적으로 점검할 수 있게 해준다.
3. 불안정성 해소
    - 코드가 완성되기 전에 반복적인 테스트를 통해 불안정성을 해소할 수 있다. 이는 결국 코드의 품질을 높이고, 사용자에게 전달되는 소프트웨어의 안정성을 보장한다.
4. 재설계 시간 절감
    - 처음부터 명확한 테스트 케이스를 작성하고 진행하기 때문에, 설계 변경이 필요할 경우 테스트를 기준으로 재설계가 진행된다. 이로 인해 불필요한 설계 변경이 방지된다.
5. 기능 추가 용이
    - 기존 코드가 이미 자동화된 테스트로 검증되었기 때문에, 새로운 기능을 추가할 때 기존 코드에 미치는 영향을 쉽게 파악할 수 있다. 테스트가 자동으로 진행되므로, 추가 기능이 기존 기능에 영향을 미치는지 빠르게 확인할 수 있다.

## TDD의 단점

1. 생산성 저하
    - TDD는 테스트 코드와 실제 코드를 동시에 작성해야 하기 때문에, 개발 시간이 전통적인 방식보다 길어질 수 있다.
    - 특히, SI(시스템 통합) 프로젝트와 같은 경우 납기일이 중요하기 때문에 TDD를 적용하기 어려울 수 있다.
2. 학습 곡선
    - TDD를 처음 도입할 때는 기존의 개발 방식과는 다른 사고방식을 요구한다.
    - 이미 익숙한 개발 패턴에서 벗어나야 하기 때문에, 기존 개발 방식에 익숙한 개발자에게는 적응하기 어렵고 시간이 걸릴 수 있다.
3. 구조에 얽매임
    - TDD를 진행하면서 테스트 코드가 원칙대로 동작하지 않거나, 예외적인 상황에서 해결 방법을 찾는 데 어려움이 있을 수 있다.
    - 이때, 원칙을 따르지 않으면 코드 품질이 떨어지기 때문에, 구조적으로 고민하는 시간이 길어질 수 있다.

---

## 5. TDD와 전통적인 개발 방식의 차이점

###  전통적인 개발 방식 (Waterfall 방식)

- 전통적인 개발 방식은 요구사항 분석 → 설계 → 개발 → 테스트 → 배포 순으로 진행된다. 
- 이 방식의 문제점은 초기 설계가 완벽하지 않다는 점이다. 
- 또한, 버그나 변경 사항이 발생할 경우 재설계가 필요하고, 이로 인해 유지보수와 코드 품질에 문제가 생길 수 있다.

### TDD 방식

- TDD는 테스트를 먼저 작성하고, 이를 통과하는 코드를 작성하는 방식이다. 
- 이 방식은 테스트 주도형 개발을 통해 지속적으로 피드백을 받을 수 있다.
- 설계가 변경되더라도 테스트 케이스에 기반해 수정이 가능하여 유연한 개발을 지원한다.

---

## 6. TDD 도구

- TDD를 지원하는 주요 도구는 JUnit이다. 
- JUnit은 Java에서 가장 널리 사용되는 단위 테스트 프레임워크로, TDD에서 가장 기본적으로 사용되는 도구이다. 
- 또한, TDD의 원칙에 따라 다양한 언어에서 xUnit 프레임워크(CUnit, CppUnit, PyUnit 등)가 등장했다.
