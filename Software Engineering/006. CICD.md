# CI/CD 개념 정리

---

## 1. CI/CD란?

- CI/CD는 소프트웨어 개발, 테스트, 배포 과정을 자동화하여 개발 생산성을 높이고, 안정적이며 빠른 릴리스를 가능하게 하는 DevOps 기반의 자동화 프로세스
- CI (지속적 통합): 코드 변경 사항을 자주 통합하고 자동으로 빌드/테스트
- CD (지속적 제공 또는 배포): 변경 사항을 자동으로 배포 가능한 상태로 만들거나 실제 배포까지 자동화

---

## 2. CI (Continuous Integration, 지속적 통합)

### 정의

- 개발자가 작성한 코드를 주기적으로 공유 리포지토리에 통합하면서, 자동으로 빌드와 테스트를 수행하는 프로세스

### 핵심 포인트

- 코드 커밋 시 자동 빌드 및 테스트
- 충돌이나 문제를 조기에 감지
- 코드의 안정성 확보

### CI 과정

1. 개발자가 feature 브랜치에 작업
2. 코드 커밋/푸시
3. CI 서버에서 자동 빌드, 테스트, Lint 수행
4. 결과 피드백 → 문제 발생 시 수정
5. main 브랜치에 merge

### 장점

- 빠른 에러 탐지 및 수정
- 코드 품질 유지
- 배포 주기 단축을 위한 기반 마련

---

## 3. CD (Continuous Delivery / Continuous Deployment)

### 공통 정의

- CI 이후 단계로, 빌드와 테스트된 애플리케이션을 자동으로 배포 가능한 상태로 유지하거나 실제 배포까지 자동화하는 것

### Continuous Delivery (지속적 제공)
- 배포 직전 단계까지 자동화
- 실제 배포는 수동 (승인 또는 버튼 클릭)
- 장점
  - 릴리스 안정성 증가
  - QA 승인 등 수동 검토 포함 가능


### Continuous Deployment (지속적 배포)
- 빌드 → 테스트 → 프로덕션 배포까지 완전 자동화
- 사람이 개입하지 않아도 사용자에게 바로 기능 제공
- 장점
  - 빠른 배포로 사용자 피드백 즉시 수용 가능
  - 배포 주기 단축, 릴리스의 일상화

## 4. CI/CD 도입 전후 비교

| 항목           | CI/CD 적용 전                             | CI/CD 적용 후                          |
|----------------|--------------------------------------------|----------------------------------------|
| 코드 통합      | 수동, 충돌 많음                            | 자동, 충돌 조기 발견                   |
| 테스트         | 수동 테스트, 종종 생략됨                   | 자동 테스트, 커밋마다 수행             |
| 배포           | 수동 배포, 시간 소모, 실수 가능성           | 자동 배포 (또는 자동화된 준비)         |
| 피드백         | 느리고 불명확함                           | 즉각적이고 구체적인 피드백 제공       |
| 생산성         | 낮음, 디버깅 시간 과다                     | 높음, 개발에 집중 가능                 |

---

## 5. 대표 CI/CD 도구 비교

| 항목           | Jenkins                                  | GitHub Actions                        |
|----------------|-------------------------------------------|----------------------------------------|
| 장점           | 오픈소스, 확장성 높음, 문서 풍부           | GitHub 통합, 설정 쉬움, 클라우드 기반  |
| 단점           | 초기 설정 복잡, 서버 필요                 | GitHub 환경에서만 사용 가능           |
| 실행 방식      | 자체 서버 또는 클라우드에서 직접 실행     | GitHub에서 자동 실행                  |
| 스크립트 형식  | Groovy 기반 (Jenkinsfile)                | YAML 기반 (workflow 파일)             |
| 적합 대상      | 복잡한 워크플로우가 필요한 대규모 프로젝트 | 빠른 구축을 원하는 스타트업/소규모 팀 |
