# 블랙박스 테스트, 화이트박스 테스트

---

## 1. 블랙박스 테스트

- 소프트웨어의 내부 구조나 코드에 대한 지식 없이, 외부 사용자 관점에서 기능성과 비기능 요구사항을 검증하는 테스트 기법이다. 
- 입력값을 제공하고 출력값을 분석하여 시스템이 예상대로 동작하는지 확인한다.

---

## 2. 블랙박스 테스트 기법

### 구문 기반 테스트 (Syntax-Driven Testing)

- 언어의 구문 규칙을 기반으로 테스트 케이스를 생성하며, 컴파일러나 문맥 자유 문법을 사용하는 시스템에 적용된다.

### 동치 분할 테스트 (Equivalence Partitioning)

- 입력값을 동등한 그룹으로 나누고, 각 그룹에서 대표값을 선택하여 테스트하여 테스트 케이스 수를 줄이면서도 다양한 입력값을 효과적으로 검증한다.

### 경계값 분석 (Boundary Value Analysis)

- 입력 도메인의 경계값에서 오류가 발생하기 쉬우므로, 경계값을 중심으로 테스트 케이스를 설계하여 오류를 발견할 확률을 높인다.

### 원인-효과 그래프 (Cause-Effect Graphing)

- 입력(원인)과 출력(효과) 간의 논리적 관계를 그래프로 표현하고, 이를 결정 테이블로 변환하여 테스트 케이스를 도출한다.

### 요구사항 기반 테스트 (Requirement-Based Testing)

- 소프트웨어 요구사항 명세서(SRS)를 기반으로 테스트 케이스를 작성하여 시스템이 요구사항을 충족하는지 검증한다.

### 호환성 테스트 (Compatibility Testing)

- 다양한 하드웨어, 운영 체제, 브라우저 등에서 소프트웨어가 정상적으로 동작하는지 확인한다.

---

## 3. 블랙박스 테스트 유형

- 기능 테스트 (Functional Testing): 소프트웨어의 기능적 요구사항을 검증한다. 
- 회귀 테스트 (Regression Testing): 기존 기능이 새로운 변경사항에 의해 영향을 받지 않았는지 확인한다. 
- 비기능 테스트 (Non-Functional Testing): 성능, 보안, 사용성 등 기능 외적인 요소를 검증한다.

---

## 4.  블랙박스 테스트 도구

- Appium: 모바일 앱의 자동화 테스트를 지원하는 오픈소스 프레임워크이다.
- Selenium: 웹 애플리케이션의 자동화 테스트를 위한 도구로, 다양한 언어와 브라우저를 지원한다. 
- Microsoft Coded UI: .NET 기반 애플리케이션의 UI 테스트를 자동화한다. 
- Applitools: 인공지능을 활용하여 UI의 시각적 요소를 자동으로 테스트한다. 
- HP QTP (QuickTest Professional): GUI 및 웹 기반 애플리케이션의 자동화 테스트를 지원한다.

---

## 5. 블랙박스 테스트의 특징

- 독립성: 개발과 독립적으로 수행되어 공정성과 중립성을 확보한다. 
- 사용자 중심: 최종 사용자 관점에서 테스트하여 실제 사용 환경을 반영한다. 
- 내부 지식 불필요: 내부 코드나 구조에 대한 지식 없이 테스트를 수행할 수 있다. 
- 요구사항 중심: 요구사항을 기반으로 테스트 케이스를 작성하여 시스템이 기대대로 동작하는지 확인한다.

---

## 6. 화이트 박스 테스트(White Box Testing)

- 소프트웨어의 내부 구조와 동작을 분석하여 테스트하는 기법이다. 
- 이 기법은 유리 상자 테스트(glass box testing), 투명 상자 테스트(clear box testing), 구조적 테스트(structural testing) 등으로도 불린다. 
- 테스터는 소스 코드에 접근하여 코드 수준에서 소프트웨어의 정확성을 검증하는 테스트 케이스를 설계한다.

---

## 7. 화이트 박스 테스트 기법

### 구문 커버리지(Statement Coverage)

- 모든 코드 구문을 최소한 한 번 이상 실행하도록 테스트 케이스를 설계한다. 
- 이를 통해 코드의 각 라인이 실행되는지 확인할 수 있다.

### 결정(분기) 커버리지(Decision/Branch Coverage)

- 각 분기점에서 가능한 모든 분기를 최소한 한 번씩 테스트한다. 
- 이를 통해 조건문에서의 모든 가능한 경로를 검증할 수 있다.

### 조건 커버리지(Condition Coverage)

- 각 조건식의 모든 가능한 결과(True/False)를 최소한 한 번씩 테스트한다. 
- 이는 결정 커버리지의 하위 개념으로, 개별 조건의 평가를 중점적으로 검증한다.


### 다중 조건 커버리지(Multiple Condition Coverage)

- 각 결정문에서 가능한 모든 조건 조합을 테스트한다. 이를 통해 복합 조건문의 모든 가능한 조합을 검증할 수 있다.

### 기본 경로 테스트(Basis Path Testing)

- 제어 흐름 그래프(Control Flow Graph)를 작성하고 사이클로마틱 복잡도(Cyclomatic Complexity)를 계산하여 독립적인 경로를 식별한다. 
- 각 독립 경로에 대해 최소한의 테스트 케이스를 설계하여 프로그램의 논리 흐름을 검증한다.

### 루프 테스트(Loop Testing)

- 프로그램 내 루프 구조의 적절한 작동을 확인한다. 
- 단순 루프, 중첩 루프 등 다양한 루프 구조에 대해 반복 횟수, 경계 조건 등을 테스트하여 루프의 정확성을 검증한다.

---

## 8. 화이트박스 테스트 도구

- JUnit : Java 애플리케이션을 위한 단위 테스트 프레임워크이다. 
- NUnit : .NET 환경에서 사용되는 단위 테스트 프레임워크이다. 
- CppUnit : C++용 단위 테스트 프레임워크로, JUnit과 유사한 구조를 가진다. 
- PyTest / unittest : Python에서 사용되는 테스트 프레임워크이다. 
- Emma, JaCoCo : Java 코드의 커버리지를 측정할 수 있는 도구이다. 
- Clover : Atlassian에서 제공하는 Java 테스트 커버리지 도구이다.

## 9. 화이트 박스 테스트의 특징

- 내부 구조 기반: 소프트웨어의 내부 코드와 논리 구조를 기반으로 테스트를 수행한다. 
- 코드 접근 필요: 테스터는 소스 코드에 접근하여 테스트 케이스를 설계하고 실행한다. 
- 정확성 검증: 코드의 각 부분이 의도한 대로 작동하는지 확인하여 소프트웨어의 정확성을 검증한다. 
- 결함 발견 용이: 내부 구조를 분석함으로써 논리적 오류나 결함을 쉽게 발견할 수 있다.
