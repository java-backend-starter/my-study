# 요구사항 분석

---

## 1. 개요

- 소프트웨어 개발에서 요구사항은 사용자가 원하는 기능과 품질을 정의
- 요구분석은 이를 정확하게 파악하는 과정
- 요구분석은 시스템의 기능을 정의하고, 요구사항을 명확히 하여 소프트웨어 개발의 성공을 보장하는 중요한 단계

---

## 2. 요구분석의 정의와 목적

- 정의: 요구분석은 사용자 요구를 조사하고 확인하여, 시스템이 제공해야 하는 기능과 품질을 명확히 정의하는 과정
- 목적: 최종 산출물인 요구분석명세서를 작성하여 개발의 기초가 되는 명확한 요구를 도출

---

## 3. 요구분석의 어려움

1. 분석가의 이해 부족: 분석가가 사용자 도메인 지식 부족 시 의사소통 문제가 발생할 수 있음
2. 사용자와 분석가 간 의사소통 문제: 사용자는 요구를 명확히 설명하기 어려워 요구가 모호해질 수 있음
3. 요구사항의 추가와 변경: 개발 진행 중 새로운 요구사항이 추가되거나 기존 요구가 변경될 수 있음
4. 모호한 요구사항: 요구가 여러 방식으로 해석될 수 있어 정확한 정리가 필요

--- 

## 4. 요구분석 절차

1. 자료 수집: 현재 시스템 문제 도출, 인터뷰 등을 통해 자료 수집.
2. 요구사항 도출: 자료를 정리하고 필요한 요구사항을 도출.
3. 문서화: 요구사항을 요구분석명세서로 작성.
4. 검증: 사용자와 요구사항을 점검하여 정확성을 확인.

---

## 5. 요구분석의 종류

1. 기능 요구사항: 시스템이 제공해야 할 구체적인 기능.
2. 비기능 요구사항: 성능, 보안, 품질 등 시스템의 품질과 제약 사항.

---

## 6. 요구분석명세서의 특성

- 완전성: 모든 기능과 비기능 요구를 빠짐없이 기록.
- 일관성: 모순 없는 일관된 요구 사항.
- 명확성: 모호한 표현 배제.
- 추적 가능성: 오류 발생 위치 추적 가능.
- 변경 용이성: 요구사항 변경이 용이.
- 검증 가능성: 시스템 검증 가능.

---

## 7. 요구분석명세서 작성 시 주의사항

- 사용자와 개발자가 이해하기 쉬운 문서 작성.
- 비기능 요구사항 명확히 기록.
- 테스트 기준으로 정량적 작성.
- 품질 우선순위 명시.

---

## 8. 요구사항 문서화 방법

- 모델링: 특정 관점으로 시스템을 단순화하여 표현.
- 모델링 언어: DFD, E-R 다이어그램, UML 등 다양한 모델링 도구 사용.
- 명세 기법:
    - 비정형 명세 기법: 자연어로 표현, 이해 쉽지만 모호할 수 있음.
    - 정형 명세 기법: 수학적 원리로 표현, 정확하지만 학습이 필요.
