# 애자일 스크럼(Scrum)과 칸반(Kanban)

---

## 1. 스크럼 (Scrum)

### 개요

- 스크럼은 애자일 방법론 중 하나로, 반복적이고 점진적인 방식으로 제품을 개발한다.
- 미식축구 용어에서 유래했으며, 팀원들이 팔을 걸고 함께 밀어붙이는 모습에서 착안했다.
- 개발을 짧은 주기(Sprint)로 나누어 반복하면서 지속적인 개선과 성장을 도모한다.

### 주요 특징

- 팀 구성: 5~9명 내외의 소규모 팀
- 개발 주기: 1~4주의 고정된 기간(Sprint)
- 산출물: 각 스프린트마다 동작 가능한 결과물 제공
- 회의:
    - 데일리 스크럼: 매일 15분 미팅
    - 스프린트 플래닝: 스프린트 시작 전 계획 수립
    - 스프린트 리뷰: 결과 공유 및 피드백
    - 스프린트 회고: 팀 프로세스 개선
- 작업 관리: 제품 백로그를 기반으로 작업 우선순위 결정
- 책임자:
    - 프로덕트 오너: 백로그 작성과 우선순위 관리
    - 스크럼 마스터: 스크럼 프로세스 지원 및 장애 제거

### 스프린트(Sprint)란?

- 의미: '전력 질주'라는 뜻으로, 집중적으로 작업을 수행하는 일정 기간
- 시작과 종료가 명확함
- 스프린트 중에는 작업 변경 금지
- 완료된 작업은 검토 후 피드백을 반영
- 매 스프린트 후 회고 진행

### 스크럼 역할 분담

- 프로덕트 오너(Product Owner)
    - 제품 백로그 작성 및 우선순위 지정
    - 목표 명확화 및 팀과 소통
    - 고객의 요구사항을 반영하며 최종 책임 담당
- 스크럼 마스터(Scrum Master)
    - 팀의 스크럼 실행을 지원
    - 장애 제거 및 스크럼 규칙 유지
- 개발팀(Development Team)
    - 실제 개발을 수행하는 자율적 팀

## 2. 칸반 (Kanban)

### 개요

- 칸반은 일본어 '간판'에서 유래하였으며, 도요타 생산방식에서 시작되었다.
- 작업 흐름을 시각화하고 지속적인 흐름(flow)을 유지하는 방식이다.
- Jira, Trello 같은 도구에서 많이 활용된다.

### 주요 특징

- 시각화: 칸반 보드 사용 (To Do → In Progress → Done 등)
- 흐름 기반: 주기 없이 작업을 흐름에 따라 지속적으로 처리
- 제한 설정: Work in Progress(WIP)를 제한하여 병목 현상 방지
- 유연성: 작업을 언제든지 추가하거나 제거 가능
- 측정: 리드 타임, 사이클 타임 등 흐름 측정 가능
- 정책: 명확한 프로세스 정책 수립과 공유

###  장점

- 전체 업무 흐름을 한눈에 파악 가능
- 병목 현상 예방 및 빠른 대응 가능
- 고정된 계획 없이 유연하게 대응 가능
- 팀 협업 및 예측 가능성 향상

## 3. 스크럼 vs 칸반 비교

| 항목       | 스크럼                    | 칸반                        |
|------------|---------------------------|-----------------------------|
| 방식       | 반복 주기 기반             | 흐름 기반                   |
| 주기       | 고정된 Sprint (1~4주)      | 주기 없음 (지속적 흐름)     |
| 계획       | 스프린트 시작 전 기획 회의 | 필요 시 작업 추가           |
| 유연성     | 스프린트 중 작업 변경 금지 | 작업을 언제든지 추가 가능   |
| 작업 제한  | 스프린트 내 작업 고정       | WIP 제한으로 동시 작업 제한  |
| 산출물     | 스프린트마다 결과물 제공   | 완료 즉시 제공              |
| 회고       | 스프린트마다 회고 실시     | 명시적 회고는 선택 사항     |

## 4. 요약

| 항목            | 스크럼                        | 칸반                         |
|-----------------|-------------------------------|------------------------------|
| 대표 도구       | Jira (Sprint 기능), Azure DevOps | Jira, Trello, Asana 등        |
| 팀 규모         | 5~9명                         | 제한 없음                    |
| 데일리 미팅     | 필수 (15분)                   | 선택 사항                    |
| 적합한 환경     | 요구사항이 명확한 프로젝트     | 잦은 요구사항 변경에 적합     |
