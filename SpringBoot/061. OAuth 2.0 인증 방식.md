# OAuth 2.0 인증 방식

---

## 1. OAuth 2.0

- OAuth 2.0은 인증이 아닌 권한 부여를 중점으로 하는 산업 표준 프로토콜로, 제3자 애플리케이션이 사용자를 대신해 특정 작업을 수행할 수 있도록 권한을 부여하는 메커니즘이다. 
- 사용자의 로그인 정보를 공유하지 않고, 애플리케이션에 접근 권한을 제공하는 방식으로, 보안과 연동성 측면에서 널리 사용된다.

---

## 2. 주요 특징

### 사용자 보호

- 사용자는 자신의 로그인 정보를 제3자 애플리케이션에 제공하지 않고, OAuth 2.0을 통해 인증된 토큰을 이용해 권한을 부여한다. 

### 연동성

- 다양한 웹 및 모바일 애플리케이션에서 활용된다.
- 예를 들어 소셜 미디어 앱이 다른 서비스와 연동하여 사진을 가져오는 등의 작업을 수행할 수 있다.

---

## 3. OAuth 2.0 주요 용어

- Resource Owner (리소스 소유자): 리소스의 원래 소유자, 보통 사용자이다. 
- Client (클라이언트): 사용자 리소스에 접근하려는 애플리케이션. 
- Resource Server (리소스 서버): 사용자의 데이터를 저장하는 서버. 
- Authorization Server (인증 서버): 인증 및 접근 토큰 발급을 담당하는 서버. 
- Access Token (접근 토큰): 클라이언트가 리소스 서버에 접근할 수 있도록 권한을 부여하는 보안 토큰. 
- Refresh Token (갱신 토큰): Access Token이 만료된 후, 새로운 토큰을 발급받기 위한 토큰.

---

## 4. OAuth 2.0 권한 부여 방식

### Authorization Code Grant (승인 코드 방식)

- 주로 웹 애플리케이션에서 사용되며, 인증 서버에서 승인을 받은 후 액세스 토큰을 발급받는 방식이다.

### Implicit Grant (암묵적 승인 방식)

- 클라이언트 사이드 애플리케이션에서 사용되며, 액세스 토큰이 URI를 통해 전달된다.

### Resource Owner Password Credentials Grant (자격 증명 승인 방식)

- 사용자가 자신의 ID와 비밀번호를 통해 인증하며, 신뢰할 수 있는 애플리케이션에서만 사용해야 한다.

### Client Credentials Grant (클라이언트 자격 증명 승인 방식)

- 클라이언트 자체 자격 증명을 사용해 액세스 토큰을 요청하는 방식입니다.

---

## 5. 장단점

### 장점

- 보안: 사용자 아이디와 비밀번호를 직접 제공하지 않으며, 제3자 애플리케이션에게 필요한 최소한의 권한만 부여한다. 
- 유연성: 다양한 애플리케이션 및 시나리오에 적합하며, 웹, 모바일, 서버 등에서 사용 가능하다. 
- 사용자 경험: 한 번의 로그인으로 다양한 서비스에 접근할 수 있어 사용자가 편리하다.

### 단점

- 복잡성: 기술적으로 복잡하여 구현이 어렵고 오류가 발생할 수 있다. 
- 보안 취약점: 잘못된 구현으로 인해 토큰 탈취 등의 보안 취약점이 발생할 수 있다. 
- 중앙집중식: 큰 서비스 제공자가 사용자 데이터를 제어하게 되어, 개인정보 보호 문제를 일으킬 수 있다.
